Denoiser loads one converted `.onnx` model per configuration (quality × HDR × aux),
fetched individually — a page never downloads the full set. Model selection is
automatic: pick options, and the library resolves the matching OIDN model name for
you.

## Quality tiers

`quality` (default `'fast'`) picks the model family size:

| `quality` | model suffix | relative cost |
|---|---|---|
| `'fast'` | `*_small` | smallest, quickest |
| `'balanced'` | *(base, no suffix)* | default-sized network |
| `'high'` | `*_large` | biggest, slowest, best quality |

Not every input combination ships all three sizes — OIDN only exports `small` and
`large` variants for some model families. When a size isn't available for your
particular combination of HDR/albedo/normal, the library silently falls back to the
base model rather than erroring.

`quality` is mutable after creation (`denoiser.quality = 'balanced'`) and takes
effect on the next `denoise()`/`denoiseTextures()` call.

## LDR vs HDR

- The image API (`denoise()`, `denoiseToFloat()`) always uses the **LDR** models —
  it expects standard 8-bit / already-tonemapped pixels.
- The texture API (`denoiseTextures()`) uses the LDR models by default, and switches
  to the **HDR** models when you pass `hdr: true`. HDR mode applies OIDN's PU
  transfer and autoexposure around the model, so you can feed it real linear HDR
  path tracer output — see [How it works](../concepts/how-it-works.mdx) for the
  pipeline detail.

## What aux buys

Pass `albedo` and `normal` alongside `color` and the library automatically switches
to the aux-guided (`*_calb_cnrm`) model — OIDN's guided models sharply improve
quality at low sample counts, because the network gets scene structure it doesn't
have to infer from noisy color alone.

```ts
await denoiser.denoise(color, { albedo, normal });
// or, on textures:
await denoiser.denoiseTextures({ color, albedo, normal, hdr: true });
```

Both aux planes are required together — the library only selects the "clean aux"
model (rasterized, noise-free guide buffers), so albedo and normal need to come from
a noise-free source such as a rasterized G-buffer, not the noisy path-traced beauty
pass. Albedo is `[0,1]` linear color; normal is encoded `[-1,1]` internally (the
image path expects RGBA8 bytes with `n*0.5+0.5` applied; the texture path takes raw
`[-1,1]` floats).

## fp16 vs fp32

`precision` (default `'fp32'`) selects the tensor/model element type:

- **`'fp16'`** — needs the device's `shader-f16` feature (falls back to fp32
  automatically if unavailable). Measured **~15% faster** than fp32, at
  **53.5–53.6 dB PSNR** vs fp32 output — visually identical.
- **`'fp32'`** — safe default, works everywhere WebGPU does.

```ts
const denoiser = await Denoiser.create({ precision: 'fp16' });
```

## File sizes

| variant | fp16 | fp32 |
|---|---|---|
| `*_small` (`quality: 'fast'`) | ~0.6 MB | ~1.3 MB |
| base (`quality: 'balanced'`) | ~1.8 MB | ~3.6 MB |
| `*_large` (`quality: 'high'`) | ~7.3 MB | ~14.7 MB |

Full set (every RT + lightmap variant, both precisions): 46 files, ~144 MB total
(96 MB fp32 + 48 MB fp16) — never downloaded at once; each page only fetches the
one file its configuration needs.

## Hosting them

Point `weightsUrl` at wherever the files live — it's plain static hosting, nothing
model-server-specific:

```ts
const denoiser = await Denoiser.create({ weightsUrl: '/models' });
```

**Production: host them yourself.** Copy the `models/` directory into your app's
static assets (or your own CDN) and pin `weightsUrl`. The default CDN URL is fine
for demos and prototypes; don't build a product on someone else's default.

**Default scheme: jsDelivr's GitHub endpoint**, version-pinned:

```
https://cdn.jsdelivr.net/gh/pmndrs/denoiser-weights@models-v2/models/<file>.onnx
```

The models are committed as ordinary git blobs (not LFS — jsDelivr can't resolve LFS
pointers) in a dedicated weights repo, served through jsDelivr's multi-provider CDN
with CORS, range requests, and no per-file limit issue (the largest file, 14.7 MB, is
well under jsDelivr's cap). GitHub Pages also works if you'd rather publish your own
branch than depend on jsDelivr.

### The `splitAux` artifacts

If you self-host, mirror the **whole** `models-v2` tag, including files named
`*.tail.onnx` and `*.enc0.bin` sitting next to the regular `.onnx` models. Here's
why: onnxruntime-web's WebGPU execution provider has a bug where the first
convolution of a model with more than 3 input channels — exactly the situation for
the 9-channel aux-guided (`*_calb_cnrm`) models — computes garbage, producing visible
speckling in the output. It's an upstream defect, not something in this library's
code; filed and reproduced at
[DennisSmolek/ort-web-webgpu-conv-bug](https://github.com/DennisSmolek/ort-web-webgpu-conv-bug).

The `splitAux` option (default **on**) works around it: the affected first
convolution is re-exported into a separate small weights file (`*.enc0.bin`) and run
in WGSL instead of through ORT, while the rest of the network runs as a `.tail.onnx`
graph through ORT as normal. This has been verified to restore output to within 1–2
LSB of native OIDN. It only applies to the 9-channel aux models — 3- and 6-channel
models are unaffected and never fetch these extra files.

If you self-host `models-v1` (or otherwise don't have the tail/enc0 files
available), the library falls back to the plain speckled model with a one-time
console warning. Set `splitAux: false` to silence the warning and force the plain
model deliberately — useful if you're waiting on a fixed onnxruntime-web release and
want to A/B the difference yourself.

### Tag policy

`models-vN` tags are immutable — once published, a tag's contents never change.
Self-hosters should mirror the exact tag their installed library version expects
(currently `models-v2`); pin it the same way you'd pin any other dependency version.

### What doesn't work as a host

Two dead ends worth knowing about so you don't re-discover them: GitHub Releases
assets send no CORS headers on the download path, so browser `fetch()` fails outright
— fine for CLIs, useless for the web. And bundling models inside the npm package
itself would balloon every `npm install` by the full ~144 MB and blow past jsDelivr's
~50 MB total-package limit, so the package would simply be refused.
